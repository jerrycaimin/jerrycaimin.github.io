JFS 客户端缓存 RDMA 配置指南

分布式缓存组（Cache Group）中启用 IORPC 与 RDMA 的完整说明
基于源码分析 · jfs @ 2026-05-27

⚡ 快速结论（TL;DR）

  • IORPC 服务端（Server）：只要加入了缓存组（--cache-group），IORPC 服务端就会自动启动，无需额外配置。
  • IORPC 客户端（Client）：必须设置 JFS_ENABLE_IORPC_CLI=1 才会启用，否则读取对端缓存仍走旧协议。
  • RDMA 支持：需要 RDMA 版本的二进制（编译时不含 nordma tag），且 JFS_RDMA_DISABLED ≠ 1。
  • --rdma-network eth0：指定 RDMA 绑定哪块网卡，若省略但二进制支持 RDMA，则使用 group IP 地址作为默认 RDMA 地址。
  • 单独指定 --rdma-network eth0 并不会自动开启客户端 IORPC，必须配合 JFS_ENABLE_IORPC_CLI=1。

🏗️1. 整体架构：IORPC 与 RDMA 的分层关系

JFS 的分布式缓存（Cache Group）依赖两个子系统：

子系统职责传输层
IORPC 节点间缓存读写的应用层 RPC 框架，取代旧的 HTTP peer 协议，支持零拷贝路径 TCP 或 RDMA（可选）
RDMA IORPC 的高性能传输层，提供绕过内核的零拷贝数据通道 InfiniBand / RoCE 硬件
ℹ️
IORPC 与 RDMA 是独立的开关：IORPC 可以跑在纯 TCP 上；只有当二进制支持 RDMA 且 RDMA 设备存在时，IORPC 才会选择 RDMA 传输。两者的客户端/服务端也是独立控制的。

⚙️2. 前置条件

2.1 二进制构建版本

JFS 使用 Go 构建标签（build tag）控制 RDMA 支持：

构建标签包含 RDMA说明
不含 nordma ✅ 是 标准 RDMA 版本，依赖 libibverbs、librdmacm
含 nordma ❌ 否 RDMAEnabled = false，所有 RDMA 代码为空桩（stub）

源码关键文件：

  • pkg/common/alloc_rdma.go build: !nordma — RDMA 内存分配器，设置 RDMAEnabled = true
  • pkg/common/rdma_stub.go build: nordma — 空实现，RDMAEnabled = false
  • pkg/rdma/rdma.go build: !nordma && linux — 实际 RDMA 实现，链接 ibverbs/rdmacm
  • pkg/rdma/stub.go build: nordma || !linux — Linux 以外的空桩
⚠️
如果当前运行的 mount 二进制不含 RDMA 支持，但指定了 --rdma-network，代码会触发 addSupport("rdma")，自动从远端下载 RDMA 版本的 mount 并替换执行（见 cmd/mount/mount.go checkSupport()）。

🔧3. 所有控制开关详解

3.1 挂载参数（mount flags）

参数类型说明
--cache-group <name> 必选 加入分布式缓存组，是启用 IORPC Server 的前提。不指定则不参与组，也不启动 IORPC。
--rdma-network eth0 可选，隐藏参数 指定 RDMA 服务端绑定的网卡名（冒号分隔多网卡）。若省略而二进制支持 RDMA，则以 group IP 作为 RDMA 监听地址。
--group-network eth0 可选 指定分布式缓存使用的网卡（用于发现 group IP，非 RDMA 专用）。
--group-port <port> 可选 指定监听端口，默认随机端口。
--cache-try-dio 可选，隐藏参数，Linux 专用 IORPC Server 启用 DIO 模式读取缓存文件（减少一次内存拷贝）。

3.2 环境变量

环境变量默认值说明源文件
JFS_RDMA_DISABLED=1 未设置（RDMA 启用） 强制禁用 RDMA，即使二进制支持 RDMA 也不使用。设为 1 时 RDMAEnabled=false，RDMA 内存池不初始化，IORPC 只用 TCP。 pkg/common/alloc_rdma.go
JFS_DISABLE_IORPC_SVR=1 未设置（Server 启用） 完全禁用 IORPC 服务端。设置后所有节点降级为旧的 HTTP peer 模式，不公告 RPC 地址。 pkg/fs/peer_rpc.go
JFS_ENABLE_IORPC_CLI=1 未设置（Client 禁用） 必须设置才能启用 IORPC 客户端。设置后，连接对端时先查询其 IORPC 地址，优先选择 RDMA 传输，否则选 TCP。版本要求 ≥ 5.2.0。 pkg/fs/peer.go
JFS_IORPC_MODE=splice pagecache IORPC Server 的数据读取模式。splice 仅在 pipe ≥ 8 MiB 且无 RDMA 时生效；其余情况均为 pagecache。设置 --cache-try-dio 后会覆盖为 dio 模式。 pkg/fs/peer_rpc.go

🔄4. 启动时的决策流程（源码逻辑）

🖥️ IORPC 服务端（Server Side）

  • 加入缓存组（--cache-group）且 NoSharing=false
  • iorpcServerMode != "disable"（默认 = pagecache）
  • 若 JFS_DISABLE_IORPC_SVR=1 → 跳过，Server 不启动
  • 若 RDMAEnabled=true：
    • 有 --rdma-network eth0 → 解析网卡 IP 作为 RDMA 监听地址
    • 无 --rdma-network → 使用 group IP 作为 RDMA 监听地址
    • 启动 Hybrid Server（TCP + RDMA 双监听）
  • 若 RDMAEnabled=false：启动纯 TCP IORPC Server

💻 IORPC 客户端（Client Side）

  • 检查 JFS_ENABLE_IORPC_CLI=1，否则直接返回（走旧协议）
  • 检查对端版本 ≥ 5.2.0，否则跳过
  • 调用 getRPCAddr 查询对端 IORPC 地址（Endpoints 列表）
  • 优先选择 RDMA endpoint：
    • 需要 RDMAEnabled=true
    • 需要本机有已初始化的 RDMA 设备（len(rdma.ListRDMADevs()) > 0）
    • 两者满足 → 创建 iorpc.NewRDMAClient
  • 否则选择 TCP endpoint → 创建 iorpc.NewTCPClient

📋5. 常见配置场景

场景 A：仅启用 IORPC（TCP 传输，不用 RDMA）

适合没有 RDMA 硬件，但希望获得 IORPC 零拷贝路径优化（sendfile / DIO）的环境。

# 所有节点挂载时加相同参数
mount --cache-group mygroup \
      --group-network eth0 \
      [其他参数...] fs /mnt/jfs

# 同时设置环境变量（所有节点一致）
JFS_ENABLE_IORPC_CLI=1   # 启用客户端 IORPC（否则读缓存仍走旧协议）
JFS_RDMA_DISABLED=1      # 明确关闭 RDMA，避免误触发
ℹ️
IORPC Server 会自动启动（不需要额外变量）。只需要 JFS_ENABLE_IORPC_CLI=1 让客户端知道要用 IORPC 协议去读对端缓存。

场景 B：完整 RDMA 路径（IORPC + RDMA 传输）

需要 RDMA 硬件 + RDMA 版本的 mount 二进制。

# 验证 RDMA 设备可用
rdma link show           # 应看到 Active 的 link

# 挂载时指定 rdma-network（也可以不指定，会用 group IP）
mount --cache-group mygroup \
      --group-network eth0 \
      --rdma-network eth0 \   # RDMA 专用网卡，也可以 eth0:eth1 多网卡
      [其他参数...] fs /mnt/jfs

# 环境变量
JFS_ENABLE_IORPC_CLI=1   # 必须：启用客户端 IORPC
# JFS_RDMA_DISABLED 不设置（默认启用 RDMA）

场景 C：只想关闭所有 IORPC（退回旧协议）

JFS_DISABLE_IORPC_SVR=1    # 禁用 Server，节点不再公告 IORPC 地址
# 不设置 JFS_ENABLE_IORPC_CLI（默认未启用），Client 自然不会用 IORPC

场景 D：DIO 模式（减少读缓存文件的内存拷贝）

mount --cache-group mygroup \
      --cache-try-dio \    # 隐藏参数，Linux only
      [其他参数...] fs /mnt/jfs

JFS_ENABLE_IORPC_CLI=1

--cache-try-dio 会将 iorpcServerMode 从 pagecache 覆盖为 dio，Server 直接用 O_DIRECT 读取磁盘缓存文件，避免 page cache 拷贝。

❓6. 关键问题解答

Q1：只指定 --rdma-network eth0，不设置 JFS_ENABLE_IORPC_CLI=1，会有 RDMA 效果吗？

❌
客户端没有 RDMA 效果。--rdma-network 只控制 IORPC Server 侧的 RDMA 监听地址。客户端（读取对端缓存）必须通过 JFS_ENABLE_IORPC_CLI=1 启用，才会在连接时查询并使用对端的 RDMA 地址。

Server 侧：RDMA 监听已启动 ✅
Client 侧：仍走旧 HTTP peer 协议，完全不经过 IORPC/RDMA ❌

Q2：JFS_ENABLE_IORPC_CLI=1 是否也会自动启用 RDMA？

ℹ️
JFS_ENABLE_IORPC_CLI=1 启用 IORPC 客户端后，会优先选择 RDMA 传输，但需同时满足：
① 本机二进制支持 RDMA（!nordma 编译）
② JFS_RDMA_DISABLED ≠ 1
③ 本机已初始化至少一个 RDMA 设备（rdma.ListRDMADevs() > 0）
④ 对端 Server 公告了 RDMA endpoint

若条件不满足，自动降级为 TCP IORPC。

Q3：IORPC Server 会自动启动吗？需要额外配置吗？

✅
是，自动启动。只要加入了缓存组（--cache-group）且 JFS_DISABLE_IORPC_SVR ≠ 1，IORPC Server 就会随挂载自动启动（默认 pagecache 模式）。无需额外变量。

Q4：--rdma-network 可以省略吗？

ℹ️
可以省略。若省略且 RDMA 二进制支持，Server 会以 <group_ip>:0 作为 RDMA 监听地址（group_ip 由 --group-ip 或 --group-network 决定）。
指定 --rdma-network 的好处是可以明确指定走哪块 RDMA 专用网卡（比如 RoCE 卡在 eth1，不是 group 通信用的 eth0），避免用错网卡。

Q5：JFS_IORPC_MODE=splice 和 RDMA 能同时用吗？

⚠️
不能同时使用。代码中 splice 模式有检查 !common.RDMAEnabled，开启 RDMA 后 splice 模式自动失效，Server 降级为 pagecache 模式。RDMA 本身已是零拷贝，不需要 splice。

✅7. 部署检查清单

纯 IORPC（TCP）部署

  • 📦
    mount 二进制无特殊要求（任意版本），建议 ≥ 5.2.0 以支持 IORPC 协议协商
  • 🔧
    挂载参数包含 --cache-group <name>（所有节点使用同一 group 名）
  • 🌍
    环境变量设置 JFS_ENABLE_IORPC_CLI=1（所有节点一致）
  • 🔥
    防火墙放通 IORPC 端口（group-port，默认随机，建议固定 --group-port 2000）
  • 📊
    查看日志 IO rpc server listened at: 确认 Server 已启动

RDMA 部署（额外检查项）

  • 🖥️
    RDMA NIC 已安装，驱动已加载（rdma link show 可见 Active 设备）
  • 📦
    使用 RDMA 版本的 mount 二进制（编译时不含 nordma），或指定 --rdma-network 触发自动下载
  • 📚
    安装系统库：libibverbs1 librdmacm1 rdma-core
  • 🔧
    挂载参数加 --rdma-network eth0（eth0 替换为实际 RDMA 网卡名，多网卡用冒号分隔）
  • 🌍
    环境变量 JFS_ENABLE_IORPC_CLI=1（启用客户端）
  • ⛔
    确认 JFS_RDMA_DISABLED 未被设置为 1
  • 🔥
    防火墙放通 RDMA 端口（端口随机，建议通过 RDMA fabric 控制面配置）
  • 📊
    查看日志中 IO rpc server listened at: 包含 rdma:// 地址
  • 📊
    查看日志中 iorpc client started, server 后出现 RDMA transport 字样
