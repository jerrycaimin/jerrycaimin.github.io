⚠ IO Timeout · 问题分析报告

JuiceFS 将 GPFS 目录设为 --cache-dir
导致 statfs 阻塞超时

当第二个 JuiceFS 客户端启动后，GPFS 触发分布式 token 协商， 导致 statfs() 系统调用挂起超过 30 秒， JuiceFS 缓存 IO 超时检测器持续输出 WARNING。 本文从源码层面剖析完整调用链和根本原因。

报告日期：2026-07-08 分析版本：JuiceFS 5.3 涉及文件：pkg/chunk/disk_cache.go 👁

第二个 JuiceFS 客户端启动后，第一个客户端开始周期性输出 WARNING， 每次间隔约 1~2 分钟，持续出现，IO 超时时间约 30 秒。

告警中的函数名 curFreeRatio.func1 是 JuiceFS 缓存管理模块中检测磁盘剩余空间的函数。

🏗背景：JuiceFS 磁盘缓存的初始化流程

执行 juicefs mount ... --cache-dir /gpfs/juicefs-cache 时， 客户端内部会调用 newCacheStore() 初始化缓存存储，并启动多个后台 goroutine 长期运行：

newCacheStore() — 初始化 在 cache-dir 下创建子目录 raw/（缓存块）和 rawstaging/（写回暂存）， 并写入 .lock 文件记录客户端 UUID，首次调用 statfs() 获取磁盘容量
go checkFreeSpace() — 每秒一次 后台 goroutine，每秒调用 curFreeRatio() 检查剩余空间/inode 比例， 决定是否触发 LRU 清理或暂停写入
go checkTimeout() — 每秒一次 专门监控 IO 操作超时的 goroutine，扫描所有正在进行的磁盘操作时间戳， 发现超过 maxIODur=30s 的操作立即告警
go checkLockFile() — 每 10 秒一次 检查 .lock 文件是否存在，用于检测缓存设备是否发生故障
go refreshCacheKeys() / scanStaging() / flush() / cleanupExpire() 扫描缓存目录中已有的文件、处理写回队列、清理过期块等
ℹ️
关键点：以上所有 goroutine 全部依赖 --cache-dir 目录上的文件系统操作。 这些操作通过统一的 checkErr() 包装执行，并同步登记到 opTs 时间戳表中。 任何一次操作超过 30 秒，checkTimeout() 都会打出 WARNING。

func (cache *cacheStore) curFreeRatio() DiskFreeRatio {
    var total, free, files, ffree uint64
    _ = cache.checkErr(func() error {          // ← 调用 ③，注册进 opTs
        total, free, files, ffree = getDiskUsage(cache.dir)  // ← 调用 ④
        return nil
    })
    ...
}

// ③ checkErr：统一 IO 包装 + 超时监控注册
// disk_cache.go:~240
func (cache *cacheStore) checkErr(f func() error) error {
    start := utils.Clock()
    cache.opMu.Lock()
    cache.opTs[start] = f   // ← 把函数指针+时间戳注册到 opTs 表
    cache.opMu.Unlock()
    err := f()              // ← 执行实际 IO，如果 GPFS 阻塞，这里会卡住
    cache.opMu.Lock()
    delete(cache.opTs, start)  // ← IO 完成后删除
    cache.opMu.Unlock()
    ...
}

// ④ getDiskUsage：底层调用 syscall.Statfs
// utils_unix.go:34
func getDiskUsage(path string) (uint64, uint64, uint64, uint64) {
    var stat syscall.Statfs_t
    if err := syscall.Statfs(path, &stat); err == nil {   // ← 被 GPFS 阻塞！
        return stat.Blocks * uint64(stat.Bsize),
               stat.Bavail * uint64(stat.Bsize),
               stat.Files, stat.Ffree
    }
    ...
}

// ⑤ checkTimeout goroutine 独立扫描超时
// disk_cache.go:272
const maxIODur = time.Second * 30   // 写死 30 秒

func (c *cacheStore) checkTimeout() {
    for c.available() {
        now := utils.Clock()
        cutOff := now - maxIODur
        c.opMu.Lock()
        for ts := range c.opTs {
            if ts < cutOff {
                // 打出上方日志中的 WARNING
                logger.Warnf("IO operation %s on %s is timeout after %s",
                    getFunctionName(c.opTs[ts]), c.dir, now-ts)
                c.state.onIOErr()
                delete(c.opTs, ts)
            }
        }
        c.opMu.Unlock()
        time.Sleep(time.Second)
    }
}
🎯
根本触发点：第 ④ 步 syscall.Statfs(path, &stat) 是导致告警的直接原因。 该系统调用在 GPFS 上被阻塞，直到 GPFS 完成内部 token 协商才返回。 期间 checkErr() 的调用方被挂起，而 checkTimeout() 检测到超过 30 秒后输出 WARNING。

🗄为什么 GPFS 作为 cache-dir 会导致 statfs 阻塞？

GPFS 是分布式并行文件系统，不是本地磁盘

GPFS（IBM Spectrum Scale）是一个多节点分布式文件系统，所有客户端节点共享同一个文件系统视图。 为了维护跨节点的一致性，GPFS 使用 Token Manager 机制： 每次访问文件或获取文件系统元数据（包括剩余空间信息），都可能需要向 Token Manager 请求令牌。

第二个客户端挂载时发生了什么？

当第二个 JuiceFS 进程启动并挂载到同一 GPFS 目录时，GPFS 需要重新协调 token 分发：

阶段GPFS 内部行为对 statfs() 的影响
新客户端注册 GPFS 检测到新的挂载客户端，开始 token redistribution 流程 暂时暂停所有客户端的元数据操作
Token 重新平衡 Token Manager 回收已分配的 token，重新向所有节点分发 Statfs() 需要拿到元数据 token 才能返回，被迫等待
Quorum 确认 多节点之间确认一致性视图 协商期间系统调用挂起，整个进程的该线程阻塞
协商完成 新 token 分发完毕，所有节点恢复正常操作 Statfs() 返回，JuiceFS 恢复正常
⚡
关键特征：GPFS 的这种阻塞是 同步阻塞（synchronous block）—— 不是 Go 层面的 channel 阻塞，而是 OS 级别的系统调用阻塞，整个 goroutine 的底层线程被挂起。 这意味着 Go runtime 的 goroutine 调度对此无能为力，只能等 GPFS 返回。 阻塞时长取决于 GPFS 集群规模和网络条件，可能数秒到数十秒。

为什么 statfs 而不是读写操作先超时？

checkFreeSpace() 每秒调用一次，是访问 cache-dir 最频繁的操作。 而实际的缓存块读写只在有 IO 请求时才触发。因此 statfs 是最先触碰到 GPFS 阻塞的操作。 若 GPFS 协商期间恰好有缓存块读写，那些操作同样会超时。

💥对业务的影响：仅 WARNING，不影响数据正确性

JuiceFS 有完整的缓存状态机（dcState）保护，单次 IO 超时不会立即让缓存失效：

WARNING 触发：checkTimeout() 打印日志 + 调用 state.onIOErr() 仅记录一次 IO 错误事件，dcState 状态机根据错误频率决定是否切换状态
GPFS 协商完成：statfs 返回，checkErr() 正常结束 此后所有 IO 操作恢复正常，state.onIOSucc() 被调用，状态机恢复 Normal
缓存继续正常工作 只要缓存状态机没有进入 dcDown 状态，缓存读写继续正常进行，数据无损
影响维度程度说明
数据正确性 无影响 JuiceFS 的数据持久化依赖对象存储，不依赖本地缓存
缓存可用性 短暂降级 GPFS 阻塞期间新的缓存写入被暂停或阻塞，但已缓存数据不受影响
读写性能 短暂下降 阻塞期间 checkFreeSpace goroutine 挂起，可能影响缓存淘汰逻辑的及时性
日志噪音 持续出现 每次 GPFS token 协商都会触发告警，多客户端场景下频繁发生
缓存彻底失效风险 存在，需关注 若超时频率过高导致 dcState 累积多次 onIOErr()，可能切换到 dcDown 状态，缓存完全禁用
🚨
潜在风险：若 GPFS 集群频繁扩容/缩容（节点加入/退出）， token 协商可能持续发生，导致缓存 IO 错误不断累积。当 dcState 状态机判断缓存设备已损坏时， 会切换到 dcDown 状态，缓存完全停用，所有 IO 直接走对象存储，性能大幅下降。

🤔为什么其他客户没有遇到这个问题？

绝大多数 JuiceFS 用户的 --cache-dir 指向本地磁盘（SSD/NVMe/HDD）， 这类文件系统的 statfs() 是纯本地调用，几乎瞬间返回（通常 <1ms）。 只有以下特殊场景才会触发类似问题：

cache-dir 类型statfs 延迟是否会超时典型用户
本地 NVMe/SSD <1ms 不会 绝大多数用户
本地 HDD 1~10ms 不会 普通用户
tmpfs / ramfs <0.1ms 不会 内存缓存用户
NFS（稳定网络） 1~50ms 通常不会 少数共享存储用户
GPFS / Lustre（分布式 FS） 可达数十秒（协商期间） 会！ HPC / 大型 AI 训练集群
Ceph FUSE 可达数秒 偶发 私有云用户
💡
触发条件唯一性：本次问题需要同时满足以下三个条件：
  1. cache-dir 指向 GPFS 挂载点（而非本地磁盘）
  2. 同一 GPFS 路径上有多个 JuiceFS 客户端（或 GPFS 客户端数量发生变化）
  3. GPFS token 协商时间超过 30 秒
只有 HPC 集群等场景下才会同时满足这三个条件，普通用户不会遇到。

为什么是第二个客户端启动时触发？

单个客户端稳定运行时，GPFS 不需要协商 token（已经稳定分发）， statfs() 响应正常。新客户端加入才是触发 GPFS token 重新平衡的条件。 这也解释了为什么第一个客户端运行时一切正常，而第二个启动后立即出现告警。

🛠解决方案与建议

❌
根本性建议：不要将网络文件系统（GPFS/Lustre/NFS/Ceph）作为 --cache-dir。 JuiceFS 的缓存设计初衷是利用节点本地高速存储（NVMe SSD）作为对象存储与计算层之间的加速层。 将缓存放在另一个分布式文件系统上，不仅引入了额外的分布式协调开销， 还与 JuiceFS 自身的分布式缓存逻辑产生冲突，得不偿失。
方案复杂度效果说明
✅ 改用节点本地 NVMe 低 根治 每个节点使用各自的本地盘，如 /mnt/nvme0/juicefs-cache，彻底避免 GPFS 依赖
✅ 使用 tmpfs 低 根治 内存充裕时用 /dev/shm 或挂载 tmpfs，statfs 纯本地无阻塞
⚠ 接受 WARNING，监控 dcState 无需改动 治标 如必须用 GPFS，监控缓存是否进入 dcDown 状态，并设置告警。WARNING 本身无损
⚠ 调整 maxIODur（改源码） 高 治标 将 maxIODur 从 30s 调大（如 120s），减少告警频率，但不解决阻塞本身
❌ 关闭缓存（--cache-size=0） 低 放弃缓存 不使用本地缓存，所有读取直接走对象存储，性能损失

📌 结论总结

  • 根本原因：GPFS 是分布式文件系统，statfs() 系统调用在 token 协商期间被 OS 级阻塞，持续超过 30 秒
  • 触发时机：第二个 JuiceFS 客户端启动导致 GPFS Token Manager 重新平衡 token 分发，期间所有客户端的元数据操作被挂起
  • 告警来源：checkFreeSpace() 每秒调用 statfs()，被 GPFS 阻塞；checkTimeout() goroutine 检测到超过 30s 后打印 WARNING
  • 业务影响：仅为 WARNING 级别日志，数据完整性无损；缓存状态机保护下，短暂超时不会立即失效；但高频超时存在缓存完全停用风险
  • 其他客户未遇到：本地磁盘的 statfs() 不存在跨节点协商，响应时间 <1ms，永远不会触发 30s 超时
  • 最优解：将 --cache-dir 改为各节点的本地 NVMe SSD，彻底避免网络 FS 的分布式协调开销
