2根因：Compaction 竞争与 OSS 404

JuiceFS 的追加写会每次产生一个新的 slice（碎片），每个 slice 是 OSS 上的独立对象。 文件碎片过多时，meta server 会调度后台 Compaction：把多个小 slice 合并成一个大对象， 释放 OSS 对象数，减少读 I/O。

完整 Race 时间线

训练任务持续追加写 → 生成大量小碎片
每次 flush/write 在 OSS 上产生一个独立对象，如 142451734_0_182（仅 182 字节）、 142532189_0_xxx 等，最终累积 10,596+ 个 slice。
🟢 Compaction Job A（先完成）
meta server 下发 REP_CHUNK_COMPACT 给某客户端，携带当时的 slice 列表，包含 142451734。
客户端：读取所有 slice → 合并写入新大块 142563040_0_283734（283 KB）。
compact2() 在 meta 更新引用：chunkIndex 0 → 142563040。
发送 REP_CHUNK_DEL → 客户端删除 OSS 对象 142451734_0_182。 chunks/16/142/142451734_0_182 → 已从 OSS 删除 ✓
⚠️ Compaction Job B（稍后调度，持有旧 slice 快照）
Job B 的 slice 列表是在 Job A 完成之前就从 meta 读取并下发的，仍包含 142451734。
调度窗口内 meta 已更新但 Job B 的快照未刷新。
🔴 Job B 读取 142451734 → OSS 404
compact(chunkid=142532189) 调用 readChunk() → readSlice(slice.chunkid=142451734)
OSS 返回 404（对象已被 Job A 删除） fail to read chunkid 142451734 [readSlice@data.go:548] readChunk() 返回 0，compact() 记录 INFO 并返回 EIO： can't compact chunk 142532189, retry later [compact@data.go:913]
🟢 meta 重新调度 Job B（使用最新 slice 引用）
下一轮调度时，meta 提供的 slice 列表不再包含 142451734（已更新为 142563040）。
Job B 成功，最终合并为 142841250（298 KB）和 142841306（2.2 MB）。
训练任务继续追加 → 新碎片 142700347, 142700348...（循环）
文件大小从 2.44 MiB 增长至 2.68 MiB，新碎片等待下一轮压缩。

竞争窗口的本质

meta server 在 compact2() 中先完成 slice 引用替换，再派发对象删除命令： 

// cmd/meta/chunks.go  compact2()
cs.replaceSlice(...)          // ① 先更新 meta 里的 slice 引用
for _, id := range old_ids {
    cs.sendDelChunk(...)      // ② 再派发对象删除（发往随机客户端）
}

但 Job B 的 slice 列表是在步骤 ① 之前就已经下发给客户端的快照， 等 Job B 真正执行时，② 已经删除了 OSS 对象 → 404。

4为什么读一次文件以后 ls/stat 也 hang？

这是与 Compaction 404 并发的第二个症状。 当用户真正的读请求（不是 compact）碰到同样的 404 时，会触发一条更长的 hang 链路： 

FUSE Read(inode, offset, size)
  └→ vfs.Read2()
       └→ ind.read0()
            └→ prepareRequests()
            └→ waitForIO()   ← 阻塞在此，等待 rreq.mode == READY
                  ↑ (300ms 轮询一次，timeout=0 时永不超时)
                  │
            [后台 goroutine] rrequest.run():
                  └→ readChunk() → readSlice(chunkid=X) → OSS 404
                  └→ n==0 → ind.trycnt++
                  └→ 等待 retry_time(trycnt) 后重试
                  └→ ...重复 ioretries 次...

retry_time 的累积延迟

trycnt单次等待累积时间
11 ms1 ms
2301 ms302 ms
3601 ms903 ms
…+300ms/次…
298,701 ms~122 s
30（最后一次）10,000 ms（上限）~132 s ≈ 2 分钟

在这 ~2 分钟里，waitForIO() 不返回 → FUSE Read 不返回 → 内核 FUSE 模块等待回复 → 内核持有该 inode 的 page lock → 任何其他进程对同一 inode 做 stat(2)/readdir(2) 需要获取 page lock → 排队阻塞。

⚠️ 这就是 cat /sys/fs/fuse/connections/3102/waiting = 2 的原因： 2 个内核 FUSE 请求（通常是 read + stat/ls）卡在等待挂载客户端回复。

~2 分钟后，ind.trycnt >= ioretries（默认 30） → rreq.done(syscall.EIO, 0) → waitForIO() 返回 → FUSE 回复 EIO → page lock 释放 → ls/stat 解除阻塞，得到 I/O error。

6数据有没有丢失？

✅ 没有数据丢失。

以下是完整推理：

当时我们非常担心数据丢失——因为不仅 ls hang 住，执行 cp /jfs/.../train.log /tmp/1.log 同样失败，文件根本无法读取。 手动执行 juicefs compact 后，文件立刻恢复正常，cp 也顺利完成。

❌ 表面看起来"丢失"
  • OSS 对象 142451734_0_182 不存在
  • 读取该 chunkid 返回 404
  • 当前 juicefs info 中看不到 142451734
✅ 实际上数据安全
  • 142451734 的 182 字节数据已被 Job A 合并进 142563040（283 KB）
  • meta 中 chunkIndex 0 已更新为 142563040
  • 读当前文件使用 142563040 → 成功

为什么手动 compact 以后就不再 hang 了？

手动 compact 将 10,594 个碎片合并成了 3 个大块（合计约 2.8 MB），meta 中的 slice 引用全部更新为新的 chunkid。此后：

  • 自动 Compaction job 读取的 slice 列表已是最新的（不再包含已被删除的旧 chunkid）
  • Race 窗口大幅收窄：job 之间重叠的 slice 数量从数千降为个位数
  • 读操作使用的 chunkid 全是有效大块 → 不再 404 → waitForIO() 正常返回 → page lock 释放 → ls/stat 不再 hang

为什么数据没有丢失？

JuiceFS Compaction 遵循 先写新 → 再更新 meta → 最后删旧 的顺序： 

// compact() 成功路径  pkg/fs/data.go:940-960
if err = writer.Finish(ctx, pos); err == nil {
    writer = nil  // 新大块写入 OSS 成功
    // 删除 block cache 里的旧条目（本地缓存）
    for _, c := range chunks {
        r := d.store.NewReader(c.Chunkid, int(c.Clen))
        g.Delete(ctx, key)   // ← 只删本地 cache，不删 OSS
    }
    // 通知 meta：compact 成功，更新 slice 引用
    server.CompactChunk(chunkid, off, chunks, nchunkid)
    // meta 的 compact2() 才会触发 OSS 对象的实际删除
}

只要 Finish() 没报错，新的 OSS 对象就一定写好了， meta 才会替换引用并删旧对象。整个过程原子性有保证， 不存在新对象未写好就删旧对象的情况。

⚠️ 但问题会复发：这个 log 文件在持续增长，compact 后不久新的碎片又会堆积， Race 条件会重新出现，ls/cp 可能再次 hang。根本解决方案见 § 8。

7不干预的话多久能自愈？

这取决于两个方向：

方向 A：Compaction 404 自愈（快）

"can't compact chunk X, retry later" 之后，meta 会在下一轮 Compaction 调度时使用最新的 slice 引用重试， 通常 几秒到几十秒内 Job B 就能成功完成（用更新后的引用）。

Compaction 本身：自愈很快，无需干预。

参考：JuiceFS 多久会自动执行一次 compact？

JuiceFS meta server 有一个持续运行的后台 goroutine 负责调度 Compaction：

  • 触发条件：当某个 chunk 的 slice 数超过阈值（默认约 50 个碎片）时，meta 立即将其加入 compact 队列
  • 调度频率：meta server 持续轮询，一旦有 chunk 进入队列就下发 REP_CHUNK_COMPACT 给某个在线 mount 客户端——无固定周期，有需求即触发
  • 每轮处理量：每个 compact job 一次最多合并约 50~200 个 slice，生成一个新大块
  • 总耗时估算：对本案例的 10,596 个 slice，假设每轮处理 100 个、耗时 1s，完整合并约需 100+ 轮 / 1~2 分钟

在这 1~2 分钟内，追加写还在持续生产新碎片，这就是"自动 compact 跑不过写入速度"的根本原因。 手动 juicefs compact 的优势在于一次性处理全部存量碎片， 而自动 compact 是分批、异步进行的，无法瞬间消灭积压。

方向 B：碎片堆积 / 新碎片持续产生（不会自愈）

如果训练任务一直在运行并持续追加写，新碎片的产生速度往往快于 Compaction 合并速度：

⚠️ 碎片堆积问题不会自愈——只要写入在继续，竞争就会持续发生， 404 日志会周期性出现，用户 I/O 会周期性 hang 几秒~2 分钟。

方向 C：单次读 hang（有上限）

假设真的是用户读请求碰到了 404（而不是 compact 路径），hang 时长有确定的上限： 

// pkg/fs/readdata.go  retry_time()
sum = Σ (i=1..29) [(i-1)*300ms + 1ms]  +  10s（第30次上限）
    ≈ 122s + 10s ≈ 132 秒（约 2 分钟）  // 默认 --io-retries=30

之后返回 EIO，调用方看到 I/O error，hang 解除。

症状自愈？时间
compact 404 报错 会 几秒~几十秒，下一轮调度时
用户读 hang 有上限 最长约 2 分钟（默认 io-retries=30），之后 EIO
碎片堆积 不会 只要追加写持续，就会持续产生
ls/stat hang 有上限 随用户读 hang 解除（同一 page lock）

8客户如何正确使用追加写 log 模式

如果客户确实需要在 JuiceFS 上持续追加写日志，以下是按优先级排列的建议：

建议 1（最有效）：减少 flush 频率，批量写入

每次 fflush() 或 fsync() 都会在 JuiceFS 层产生一个新 slice。 训练框架如果每行/每 step 都 flush，一分钟可能产生数百个碎片。

改为按时间（如每 5s）或按大小（如 1 MB）批量刷盘：

# Python 示例：用行缓冲替代即时 flush
import logging, sys

handler = logging.StreamHandler(sys.stdout)
handler.setLevel(logging.DEBUG)

# 不要每行 flush，改为按时间 flush（如 logging.handlers.TimedRotatingFileHandler）
# 或者写到内存 buffer，定期 flush

每个 slice ≥ 1 MB 时，Compaction 压力大幅降低，Race 概率接近零。

建议 2：写到本地盘，任务结束后再拷贝到 JuiceFS

训练过程中日志写到本地 /tmp 或 hostPath，任务结束/checkpoint 时 cp /tmp/train.log /jfs/.../train.log。 这样 JuiceFS 对象是一次性写入的大文件，没有碎片问题。

建议 3（缓解）：定期手动或 cron compact

如果业务无法改变写入模式，可以设置定期 compact：

# 每 10 分钟对日志目录做一次 compact
*/10 * * * * juicefs compact /jfs/oss-import/zachytong/sipai/.../logs/ --recursive

compact 后碎片归零，Compaction Race 概率降为零，直到下一波碎片堆积。

建议 4（缓解 hang 时长）：调小 --io-retries

默认 --io-retries=30 导致单次读 hang 最长 ~2 分钟。 改为较小值可显著减少 hang 时长：

--io-retries最大 hang 时长适用场景
30（默认）~132 秒OSS 偶发抖动，需要高容忍
10~13 秒对延迟敏感，能接受偶发 EIO
5~3 秒快速失败，业务自行重试
⚠️ 注意： 调小 --io-retries 不能解决 404 根因， 只是让应用更快看到 EIO，需要应用层做好重试。

建议 5（研发层面）：meta 调度改进

根本解法是 meta server 在下发 Compaction job 时，给每个 slice 加版本戳或 epoch， 客户端收到 404 后能判断"这个 slice 已被上一个 compact 合并，我直接用新引用"， 而不是 retry+EIO。这需要研发侧改动 Compaction 协议。

总结建议优先级

优先级方法效果改动成本
P0 减少 flush 频率（>=1MB/flush） 从根本消除碎片，Race 概率接近零 业务改代码（低～中）
P1 日志写本地，完成后拷 JuiceFS 完全消除碎片 架构调整（中）
P2 定期 cron juicefs compact 控制碎片数量上限 运维脚本（低）
P3 调小 --io-retries 缩短单次 hang 时长 挂载参数（极低）
P4 研发改进 Compaction 协议 彻底消除 Race 副作用 内核研发（高）
