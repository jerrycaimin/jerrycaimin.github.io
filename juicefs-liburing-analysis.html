① libaio 与 liburing 是什么

两者都是 Linux 上的异步 I/O（AIO）方案，目标是：提交 I/O 请求后不阻塞，程序继续干别的事，等 I/O 完成再通知。

libaio —— 老牌 Linux AIO（2000 年代初）

// 工作流程：每次操作 = 一次系统调用
io_submit(ctx, nr, iocbpp);    // 提交（系统调用）
io_getevents(ctx, min, nr, events, timeout); // 收割结果（系统调用）
关键限制：只对 O_DIRECT（直接 I/O）真正异步。使用 page cache（普通 open）时退化为同步阻塞，io_submit 直接卡住。

liburing —— 新一代 io_uring（Linux 5.1+，2019）

核心创新：内核与用户态之间建立两个共享的环形队列（Ring Buffer），通过内存映射（mmap）直接通信，可完全绕过系统调用。

// SQ Ring（提交队列）← 用户写 SQE（请求描述符）
// CQ Ring（完成队列）← 用户读 CQE（完成事件）
// 内核直接读写两个 Ring，无需数据拷贝

struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, len, offset);
io_uring_submit(&ring);       // 通知内核（SQPOLL 模式下可省略）

struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe); // 等结果
维度libaioliburing (io_uring)
提交机制每次 io_submit = 一次系统调用共享环形队列，SQPOLL 模式可 0 系统调用
真正异步条件必须 O_DIRECT缓冲 I/O 也支持（io-wq 兜底）
支持操作类型仅文件 I/O文件 + 网络 + 管道 + 定时器等
现状逐渐被取代Linux 5.1+ 推荐方案

② submit 和 reap 是什么

io_uring 的使用分两个阶段：

阶段一
Submit（提交）
用户把 I/O 请求写入 SQ Ring，调用 io_uring_submit() 通知内核
→
阶段二
Reap（收割）
内核 I/O 完成后写入 CQ Ring，用户通过 io_uring_peek_cqe() 等读取结果
  • submit 慢：提交请求到内核这一步卡住了
  • reap 慢：I/O 实际完成、把结果拿回来这一步慢（正常，是真正的等待）

③ 客户现象：本地 vs 云上的 submit/reap 分布

🖥 本地集群（本地 NVMe）
submit 耗时快（µs~ms）
reap 耗时相对慢
☁️ 云上（云盘 / 对象存储）
submit 耗时慢（ms~s 级）
reap 耗时几乎立即

根因：io_uring 的 inline execution（内联执行）机制

io_uring 在 io_uring_enter（submit 系统调用）里，会先尝试在提交线程的上下文里同步执行请求，而不是立即交给 io-wq 后台线程。

io_uring_submit()
  └─ 内核 io_uring_enter()
       │
       ├─ 尝试 inline 执行请求
       │    │
       │    ├─ 底层操作返回 -EAGAIN（暂时无法完成）
       │    │     → 丢给 io-wq 后台线程异步重试
       │    │     → submit 立即返回 ✓（本地 NVMe 场景）
       │    │
       │    └─ 底层操作同步阻塞完成
       │          → inline 在提交线程上等完
       │          → submit 耗时 = 实际 I/O 时间（云盘高延迟 → s 级）
结论：不同存储系统（本地盘 vs 云盘）的底层驱动返回值不同，io_uring 选择了不同的执行路径——本地盘驱动返回 -EAGAIN，走 io-wq 异步；云盘驱动直接同步阻塞，耗时全部计入 submit。这正是客户观察到「云上 submit 慢、reap 快」的根本原因。

submit 完成时序对比

== 本地 NVMe（走 io-wq）==
submit ──▶ io-wq 线程接手 ──▶ submit 立即返回
                    ↓
              NVMe I/O 完成 ──▶ 写 CQE ──▶ reap 拿结果（稍慢）

== 云盘（inline 同步执行）==
submit ──▶ 内核 inline 等云盘 I/O 完成（高延迟卡住）──▶ 写 CQE ──▶ submit 才返回
                                                                 ↓
                                                          reap 几乎立即完成

④ 优化方案

根本目标：无论云上还是本地，都让 submit 快速返回，I/O 真正异步化。

1
强制异步 IOSQE_ASYNC（最直接）
给每个 SQE 设置该 flag，跳过 inline execution，直接丢给 io-wq 后台线程。
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, len, offset);
sqe->flags |= IOSQE_ASYNC;  // 强制异步
io_uring_submit(&ring);      // 立即返回
效果：高
2
SQPOLL 模式（消除 submit 系统调用）
内核起专属线程轮询 SQ Ring，用户写完 SQE 无需调用 io_uring_submit()。
struct io_uring_params params = {};
params.flags |= IORING_SETUP_SQPOLL;
params.sq_thread_idle = 2000; // 2s 后休眠
io_uring_queue_init_params(DEPTH, &ring, ¶ms);
代价：SQ 线程持续占一个 CPU 核，适合高吞吐场景。
效果：高（高负载）
3
批量提交 + 批量收割
攒多个 SQE 一次 submit，减少系统调用次数，摊薄单次开销。
// 一次提交多个请求
for (i = 0; i < BATCH_SIZE; i++) {
  sqe = io_uring_get_sqe(&ring);
  io_uring_prep_read(sqe, fds[i], bufs[i], len, off[i]);
  sqe->flags |= IOSQE_ASYNC;
}
io_uring_submit(&ring);
// 批量收割
n = io_uring_peek_batch_cqe(&ring, cqes, BATCH_SIZE);
效果：中
4
提交线程与收割线程分离
submit 线程即使被云盘阻塞也不影响主业务逻辑；收割线程独立处理完成事件。
// 生产线程 → 提交线程（可能阻塞）
//                      ↓
//             收割线程（独立拿结果）
pthread_create(..., submit_thread, &ring);
pthread_create(..., reap_thread, &ring);
效果：中（架构调整）

增大队列深度（云盘专项）

云盘延迟高但带宽大，适合增大 inflight 请求数，让 I/O 充分流水线化：

// 云上可将队列深度从默认 64 提升到 256~1024
io_uring_queue_init(512, &ring, 0);

⑤ JuiceFS（FUSE 文件系统）场景的特殊性

JuiceFS 的 I/O 路径

应用进程
   │  io_uring / read / write
   ▼
VFS（内核虚拟文件系统层）
   │
   ▼
FUSE 内核模块  ←── io_uring 只能影响到这里
   │  /dev/fuse 通信
   ▼
JuiceFS 用户态进程（Go 程序）
   │  HTTP 请求
   ▼
对象存储（S3 / OSS / 阿里云 OSS ...）
注意：JuiceFS 本身支持 O_DIRECT。本次客户场景是任务代码没有使用 O_DIRECT，所以 libaio 走了缓冲 I/O 同步路径，io_uring 的 inline execution 也受此影响。

各优化方案在 JuiceFS 上的效果

优化方案在 JuiceFS/FUSE 上的效果原因
IOSQE_ASYNC 部分有效 submit 不卡了，但 io-wq 线程仍阻塞在 FUSE 操作上，只是转移了阻塞点
SQPOLL 模式 基本无效 内核 SQ 轮询线程执行 FUSE 操作时照样阻塞
O_DIRECT 有效（JuiceFS 支持） 绕过 page cache，避免缓冲 I/O 同步路径，需要应用层显式开启
多线程并发 io_uring 有效 多个 io-wq 线程并行阻塞，聚合吞吐上去
批量提交 + 大队列深度 有效 多请求同时 inflight，流水线化
调 JuiceFS 挂载参数 最有效 直接作用于 JuiceFS 内部并发层，效果最直接

JuiceFS 关键挂载参数

juicefs mount ... \
  --buffer-size=4096     # 读写缓冲区大小（MB）
  --max-uploads=50       # 上传并发数
  --max-downloads=50     # 下载并发数
  --prefetch=10          # 预读并发
结论：io_uring 的优化对 JuiceFS 这类 FUSE 文件系统帮助有限。最简单有效的组合是：

1. 应用层开启 O_DIRECT（如果任务允许）
2. 给所有 SQE 加 IOSQE_ASYNC flag，先确认 submit 是否变快
3. 调 JuiceFS 自身的并发参数（--max-downloads 等），收益最直接
