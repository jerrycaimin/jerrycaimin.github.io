2Meta 主循环 CPU 100% 的真正原因

⚠️ 注意区分：主循环打满 ≠ 整个节点 CPU 打满

JuiceFS meta 的主循环（dispatch goroutine）是单线程的事件循环，它处理所有客户端请求。当主循环被某个耗时操作阻塞时，就会表现为「主循环 CPU 100%」， 其他 goroutine（如 TCP send、Raft 复制）不受影响，所以整个节点的 CPU 可能是 172%（多核利用），但主循环只使用 1 个核。

为什么内存只用了 60% 还会触发 zstd 压缩？

根因在 filesystem.go:843：

ArchiveMinTime = cfg.getuint32("ARCHIVE_MINTIME", 600)

Archive（归档压缩）机制的触发条件不是内存不足，而是时间！
ARCHIVE_MINTIME 默认值是 600 秒（10 分钟）， 任何超过 600 秒未被访问的 inode 内存页都会被 zstd 压缩并从活跃内存中移出。 这个行为在进程启动后就一直运行，与内存用量无关。

只有显式设置 ARCHIVE_MINTIME=0 才能完全禁用。内存低于 15% 时，系统会加速触发， 但在 wl-meta1 当前 60% 内存使用率下，它按正常节奏运行，每轮主循环（main_eachloop）都会调用一次 fs_try_archive()。

主循环阻塞的完整机制

①
主循环每轮调用 fs_try_archive()
遍历 restored[] 列表，对冷 inode 调用 compressPage()
②
zstd 压缩是 CGO 调用，单次可阻塞 1-10ms
1.2 亿 inode 中大量为"冷"数据，积累了很多待压缩的页面（pages 计数）
③
客户端访问触发 decompressPage()（解压）
每次 OPEN/LOOKUP 访问已归档的 inode 都需要先解压：fsnodes_search_archived → fsnodes_restore → decompressPage
④
解压后的 inode 加入 restored[]，等待再次被压缩
4700 ACQUIRE/s 不断访问 344 万个文件，形成 decompress → serve → re-compress 的持续循环
⑤
41 个 goroutine 卡在 main.send（msg.go:255）
主循环偶尔批量释放响应，41 个 TCP send goroutine 同时争抢内核发送缓冲区 → 全部阻塞在 conn.Write()

💡 关键结论：41 个 goroutine 卡在 main.send 是结果，不是原因。 真正的原因是 zstd CGO 调用阻塞了主循环，导致积累了大量响应同时涌出。

3Changelog 里为何有如此多的 ACQUIRE 操作

ACQUIRE 是什么？

ACQUIRE(session_id, inode) 是 JuiceFS meta 用于追踪"文件被打开"状态的 Raft changelog 条目， 由客户端调用 open() 系统调用时触发（Linux SYS_openat）。 

# 触发路径（源码 openfiles.go）
客户端 open() syscall
  → meta REQ_OPEN 请求
  → server.go: fs_opencheck() 检查权限
  → openfiles.go: of.open(sid, inode)
    → 首次打开该 session 下的 inode：
      log.gen(nil, "ACQUIRE(%d,%d)", sid, inode)
      → changelog.go: gen() → raft.send_entry()
      → 写入 Raft log，复制到 Follower
      → 写入本地 .jfs 文件

# REQ_SUSTAINED_INODES 心跳（每 10 秒一次）
of.sync(sid, inodes)
  → 计算 diff（新打开的文件）
  → 批量写入：ACQUIRE(sid, inode1,inode2,...) 最多 1024 个/行

设计意图： Raft changelog 记录 ACQUIRE 是为了让 Follower 同步「哪些文件正在被打开」 （ofStore），这样 Leader 切换后新 Leader 也知道哪些 inode 不能被垃圾回收。

为什么会有如此多 ACQUIRE？

指标实测数据来源
活跃客户端 15 台训练节点（session ID 998-1047） changelog grep
ACQUIRE TPS 约 4700 次/秒 changelog 行数 ÷ 时间跨度
每小时唯一 inode 数 约 344 万 每小时 changelog 去重统计
相邻小时 inode 重叠率 90.2% hour N 与 hour N-1 的 inode set 交集
1 小时 changelog 产生量 约 1.5 GB（69.1 MB/分钟） 当前 changelog 文件大小
非 ACQUIRE 操作占比 <0.1%（RELEASE 13611, ACCESS 5347, CHUNKDEL 236） grep -v ACQUIRE | wc -l

90.2% 重叠率意味着什么？

同样的 344 万个文件，每小时被扫描一遍，周而复始。
这 344 万个 inode 约占 1.2 亿总 inode 的 2.9%， 说明训练节点在反复读取同一批文件（训练 dataset 的 epoch 式循环或 prefetch 池）。

每次 open() → ACQUIRE：即使文件已经在 ofStore 里， 如果 session 重连（meta 重启后 ofStore 清空），所有文件要重新 ACQUIRE，形成洪峰。

4完整的因果链（Causal Chain）

以下是从业务行为到 CPU 打满再到 meta 重启的完整因果链。红色节点为关键故障点。

🔵
【业务根源】15 台训练节点持续扫描 ~344 万个文件（训练数据 epoch 循环）
每个 open() 系统调用都经由 FUSE → JuiceFS 客户端 → meta REQ_OPEN → ACQUIRE
↓
⚡
每秒约 4700 个 ACQUIRE 写入 Raft Changelog
每个 ACQUIRE = 1次 meta.version++ + 1次 Raft 日志复制（需要 2/3 节点确认） + 写入 .jfs 文件
↓
⚡
Changelog 快速堆积：1 小时约 1.7 亿 version 增量，产生 ~1.5 GB 文件
当前实测：changelog.1614850000-17.jfs 已达 691 MB，11.5M 行
↓
⚡
1.2 亿 inode 元数据常驻内存（活跃服务节点 16 GB）
Archive 机制（ARCHIVE_MINTIME=600s，默认开启）将超过 10 分钟未访问的 inode 用 zstd 压缩。
344 万个被频繁访问的 inode 不断地被 restore（解压）又被 archive（压缩）。
↓
🔴
【CPU 打满根因】zstd CGO 调用阻塞主循环
CPU profile 实测：compressPage 占 28.67%，decompressPage 占 12.46%，合计 41% CPU 全部是 zstd。
主循环中 compressPage 每次 1-10ms → 主循环延迟上升 → 请求队列积压 → 表现为"主循环 CPU 100%"
↓
🔴
【wl-meta3 特有】8 GB 内存不足 → 内存 < 10% → archive 激进触发 → OOM
wl-meta3 是 8 GB 冷备节点，曾短暂被选为 Leader（网络分区期间）。
8 GB 内存 + 1.2 亿 inode + changelog 堆积 → 内存低于 10% 触发更激进的压缩和重启机制 → 重启 17 次。
重启后 ofStore 清空 → 15 个客户端重连 → of.sync() 批量生成 ACQUIRE 洪峰 → 加速下一次重启
↓
🟣
当前状态（wl-meta1 Leader，16 GB）：CPU 172%，无 OOM，但响应延迟升高
内存充裕不会 OOM，但 zstd compress/decompress 循环持续消耗 CPU。
4700 ACQUIRE/s → changelog 写入 → Raft 复制 IO → TCP send goroutine 积压。
🔑 一句话总结

训练节点每秒 4700 次文件 open() → 4700 ACQUIRE/s → 344 万 inode 反复 decompress/compress（zstd CGO）→ 主循环被 zstd 阻塞 → CPU 打满。wl-meta3 的 OOM 是同一问题在 8 GB 内存下的极端形态。

5下一步调查与优化建议

💼 业务层面（找业务方确认）

  • 确认扫描模式：15 台训练节点是否在做 epoch 循环？ 是否每轮都需要 open() 全量文件，还是可以保持文件句柄不关闭？
  • 减少 open/close 频率：如果训练框架（PyTorch DataLoader 等）每个 step 都重新打开文件， 建议改用 mmap 或复用文件描述符，可将 ACQUIRE 频率降低 10x。
  • 调查实际 IO 模式：使用 strace -p <pid> -e trace=openat,read,close 抓取训练进程的实际系统调用，确认每秒真实 open 次数。
  • 是否需要列举所有文件：部分场景用 readdir + 内存缓存替代每次 open， 只在实际读数据时才 open，可大幅减少 ACQUIRE。
  • 确认 344 万文件是一批还是滚动：如果是固定的训练集，考虑提前 prefetch 到本地 SSD， 完全绕开 JuiceFS 的 open 开销。

🔧 运维层面（可以立即操作）

  • 【最有效】升级 wl-meta1/2 内存到 32 GB+： 当前 16 GB 在 1.2 亿 inode 下只有 ~40% 余量，archive 触发频繁。 升级到 32 GB 后 archive 压力减半，zstd CPU 开销将明显下降。
  • 调大 ARCHIVE_MINTIME：在 meta.cfg 中设置 ARCHIVE_MINTIME=3600（1 小时），减少压缩频率。 但这会增加内存用量，需评估是否会超过 16 GB。
  • 彻底禁用 archive（谨慎）：设置 ARCHIVE_MINTIME=0， 让所有 inode 常驻内存。需确保 16 GB 足以容纳所有活跃 inode（当前约 8.5 GB，有余量）。
  • 确保 wl-meta3（8 GB）永不成为 Leader： 检查 Raft priority 配置，确认 wl-meta3 的 priority 低于 wl-meta1/2， 避免网络波动时 8 GB 节点被选为 Leader 导致 OOM 重启循环。
  • 监控 ACQUIRE TPS：在 meta 的 Prometheus metrics 中监控 ACQUIRE/s， 当 >3000 时告警。
