问题分析报告

JuiceFS CSI + Docker-in-Docker
挂载路径 mkdir /xxx: permission denied

业务 Pod 内运行 Docker 启动子容器，bind mount JuiceFS 挂载路径时报错。 本文从 CSI Driver 源码层面解释根本原因，并提供完整解决方案。

📅 2026-07 🔧 juicefs-csi-driver 🐳 Docker-in-Docker 组件：JuiceFS CSI Driver（Mount Pod 模式） 👁

1报错现象

❌
错误信息
error while creating mount source path '/cloud/data/inference_logs/.../prometheus.yaml': mkdir /cloud: permission denied

场景描述：在业务 Pod 内通过 Docker（Docker-in-Docker）启动 Prometheus 容器，并将 JuiceFS PVC 挂载路径下的配置文件作为 bind mount source 传入。

  • 业务 Pod 内访问 /cloud/data/... 路径：✅ 正常，文件可读
  • Pod 内 Docker daemon 启动子容器，bind mount 同一路径：❌ mkdir /cloud: permission denied

2JuiceFS CSI 的挂载链路

JuiceFS CSI Driver 采用 Mount Pod 模式，挂载路径从 FUSE 进程一路传播到业务 Pod，经历以下三层：

┌──────────────────────────────────────────────────────────────────┐
│                        Node（宿主机）                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Mount Pod (juicefs-xxx)                                 │    │
│  │  JuiceFS FUSE 进程                                       │    │
│  │  VolumeMount: MountPropagation = Bidirectional  ←────────┼──── pod.go:295
│  │  /jfs/pvc-xxx  →  传播到宿主机                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                         │                                        │
│       宿主机路径: /var/lib/juicefs/pvc-xxx  (FUSE 可见)          │
│                         │                                        │
│    BindTarget() bind mount  ←──────────────────── node.go:238   │
│                         │                                        │
│  ┌──────────────────────▼──────────────────────────────────┐    │
│  │  Business Pod                                            │    │
│  │  /cloud  ← bind mount from kubelet                       │    │
│  │  mountPropagation: ??? (默认 None = private)  ←── ⚠️关键  │    │
│  │                                                          │    │
│  │  Docker daemon (DinD)                                    │    │
│  │    └─ 新建 mount namespace                               │    │
│  │         └─ 看不到 /cloud (private 不继承) ← ❌ 根本原因   │    │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘

关键代码位置

文件内容
pkg/juicefs/mount/builder/pod.go:295 Mount Pod 自身的挂载点设置为 MountPropagationBidirectional，保证 FUSE 挂载传播到宿主机
pkg/driver/node.go:238 jfs.BindTarget() 将宿主机 FUSE 路径 bind mount 到业务 Pod 的 target path
pkg/juicefs/juicefs.go:197 fs.Provider.Mount(bindSource, target, "none", []string{"bind"}) 执行实际 bind

CSI Driver 本身的链路完全正确，问题不在 JuiceFS 侧。

3根本原因：Mount Namespace 传播语义

Linux Mount Propagation 三种模式

Kubernetes 配置值 Linux 内核属性 新建子 namespace 能看到该挂载吗？
None（默认） private ❌ 不能
HostToContainer slave ✅ 能（宿主机 → Pod，单向）
Bidirectional shared ✅ 能（双向，需要特权）

DinD 场景下的触发机制

  1. 业务 Pod 的 JuiceFS volumeMount 未设置 mountPropagation，默认为 None（Linux private）
  2. Pod 内 Docker daemon 启动 Prometheus 子容器时，调用 clone(CLONE_NEWNS) 创建新的 mount namespace
  3. 由于父 namespace（Pod）中 /cloud 是 private，新 namespace 继承不到这个 FUSE 挂载点
  4. Docker daemon 在新 namespace 视角中 /cloud 不存在，走入 createMountSourcePath() 逻辑
  5. os.MkdirAll("/cloud/data/.../prometheus.yaml") → 第一步 mkdir /cloud → Pod rootfs 根目录无写权限 → permission denied
💡
验证方式
在业务 Pod 内执行 cat /proc/self/mountinfo | grep cloud，确认 /cloud 存在。 再进入 Pod 内 Docker 容器（docker exec）执行同样命令，如果看不到 /cloud，就确认了这个问题。
⚠️
另一种可能（FUSE allow_other）
如果 Docker daemon 以非 root 用户（或 user namespace 重映射后的 uid）运行，而 JuiceFS FUSE 未开启 allow_other， 则 lstat 直接返回 EACCES，Docker 同样会误判路径不存在并触发 mkdir。

4解决方案

✅ 方案一：设置 mountPropagation（推荐，最简单）

在业务 Pod 的 volumeMount 中显式声明挂载传播模式：

spec:
  containers:
  - name: your-container
    volumeMounts:
    - name: juicefs-pvc
      mountPath: /cloud
      mountPropagation: HostToContainer  # 加上这一行
✅
HostToContainer（slave）通常已足够：宿主机的 FUSE 挂载传播进 Pod 的所有子 namespace，包括 DinD 新建的 mount namespace。 Bidirectional（shared）需要容器具备 SYS_ADMIN 权限，一般 DinD 场景不需要双向传播。

✅ 方案二：JuiceFS 挂载时开启 allow_other

如果怀疑是 FUSE 权限导致，在 PV 或 StorageClass 的 mountOptions 中加入：

mountOptions:
  - allow_other

这允许非挂载所有者的进程（包括 DinD 里的 Docker daemon）访问 FUSE 文件系统。

✅ 方案三：规避文件级 bind mount

如果是将 JuiceFS 上的 prometheus.yaml 直接 bind mount 进 Docker 子容器，可以改为：

  • 在启动脚本中先 cp /cloud/.../prometheus.yaml /tmp/prometheus.yaml，再 bind mount /tmp 路径
  • 或将配置文件独立成 ConfigMap，不依赖 JuiceFS 路径

推荐排查顺序

步骤操作预期
1业务 Pod volumeMount 加 mountPropagation: HostToContainer大概率解决
2如未解决，检查 JuiceFS mountOptions 是否有 allow_other解决 FUSE 权限问题
3检查 DinD Docker daemon 是否以特权模式运行（--privileged）特权模式下 mount propagation 更宽松

5总结

✅
这不是 JuiceFS CSI Driver 的 Bug
JuiceFS 的 FUSE 挂载传播链路（Mount Pod → 宿主机 → 业务 Pod bind mount）完全正确。 问题发生在业务 Pod 内，DinD 新建 mount namespace 时无法继承 private 的 FUSE 挂载点。
现象根本原因修复方式
mkdir /xxx: permission denied 业务 Pod volumeMount 未设 mountPropagation，DinD 新 namespace 看不到 FUSE 挂载 volumeMount 加 mountPropagation: HostToContainer
Pod 内访问正常，Docker 子容器访问失败 Pod container 和 Docker container 在不同 mount namespace，private mount 不继承 同上；或开启 FUSE allow_other
