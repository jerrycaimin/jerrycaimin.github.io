1volumeDevices vs volumeMounts 是什么

Kubernetes 里 PVC 有两种完全不同的使用方式，取决于 PV 的 volumeMode：

volumeMode: Filesystem volumeMode: Block
Pod 里使用方式 volumeMounts（挂载为目录） volumeDevices（暴露为裸块设备）
容器看到什么 /mnt/cache（可以 ls、open()） /dev/xvdb（裸设备，没有文件系统）
谁来格式化 PV 创建时 K8s 已格式化好 容器自己处理，或应用直接 raw I/O
典型使用者 几乎所有应用、JuiceFS 缓存 数据库（PostgreSQL/MySQL 直写）、Ceph OSD

Filesystem 模式的完整流程

①
ACS 格式化本地 NVMe 盘（ext4/xfs）
由 StorageClass provisioner 完成，用户无感知
↓
创建 Local PV，带 nodeAffinity 绑定到该 Node
PV 只能被该 Node 上的 Pod 使用
↓
PVC 绑定 PV，状态变为 Bound
↓
Pod 通过 volumeMounts 挂载 PVC
容器内得到 /mnt/cache 目录，可直接读写文件

Block 模式（local-passthrough）的流程

①
ACS 将裸块设备直通给容器，不做任何格式化
↓
Pod 通过 volumeDevices 使用 PVC
↓
容器内得到 /dev/xvdb，一个裸块设备
↓
应用必须自己做 raw I/O 或先 mkfs + mount
普通业务（AI 训练、Python 脚本）完全不知道怎么用裸块设备
⚠️ 关键约束： Kubernetes 明确禁止同一个 volume 同时出现在 volumeMounts 和 volumeDevices 中。 Block 模式 PVC 只能用 volumeDevices，JuiceFS Operator 的 cacheDirs 只生成 volumeMounts， 二者根本无法共存。

2Operator 默认调度 vs enableScheduling

这里有个很容易混淆的点：Operator 的两种调度模式行为完全不同。

默认模式（不开 enableScheduling）

Operator 自己实现了类似 DaemonSet 的逻辑，完全不走 K8s 调度器：

→
扫描所有打了 juicefs.io/cg-worker: "true" 标签的 Node
→
对每个 Node 创建一个 Worker Pod，直接指定 nodeName
→
天然一个 Node 一个 Worker，不需要反亲和性配置

对于 type: HostPath 的场景，默认模式完全够用，YAML 极简。

enableScheduling: true 模式（v0.7.2+）

调度权交给 K8s 标准调度器，好处是可以使用完整的亲和性规则（跨 zone 分布等），代价是：

  • 调度器默认不知道"一个 Node 只能一个缓存 Worker"，可能把多个 Worker 塞到同一台节点
  • 必须手动配置 podAntiAffinity 才能保证每台 Node 最多一个 Worker
  • 对于 Local PVC，调度器才能感知 PV 的 nodeAffinity，避免 Pod 被调度到没有对应 PVC 的节点
✅ HostPath 场景

不需要 enableScheduling

默认模式天然一 Node 一 Worker，配置最简单

⚠️ Local PVC 场景

需要 enableScheduling: true

让调度器感知 PV 的 nodeAffinity，但随之要手写反亲和性规则

反亲和性（podAntiAffinity）的含义

podAntiAffinity 是告诉调度器：我不想和同类 Pod 在一起。 加上 topologyKey: kubernetes.io/hostname 后，效果就是同一个缓存组的 Worker Pod 每台 Node 最多一个： 

affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:   # 硬性要求，不满足就不调度
      - labelSelector:
          matchExpressions:
            - key: juicefs.io/cache-group
              operator: In
              values: [my-cache-group]
        topologyKey: "kubernetes.io/hostname"   # 以 Node 为粒度隔离

3ACS local-passthrough 的根本矛盾

阿里云 ACS 提供了一种 local-passthrough StorageClass，将 PPU 本地盘以 Block 模式直通给 Pod。 这是为数据库等存储感知型应用设计的高性能方案，但对于 JuiceFS 场景存在根本性的矛盾。

"就是说现在 ACS 提供的这个 PVC 是 local-passthrough（Block 模式），也就是业务 Pod 拿到的是一个 /dev/xvdb 而不是一个路径对吧，那就有点扯了，业务要一个 block device 干嘛呀，要一个路径挂载点呀。operator 是不支持 volumeDevices，也没必要，支持了用户也用不了，难道还给你格式化一下再用么 😅"
根本矛盾：
  • local-passthrough 只支持 Block 模式 → 只能用 volumeDevices
  • JuiceFS Operator 的 cacheDirs 自动生成 volumeMounts
  • K8s 禁止同一 volume 同时出现在两者中 → 直接报错，无法绕过

为什么 AI 训练业务也不能直接用 Block PVC？

普通的 AI 训练任务、Python 脚本的诉求是：open("/data/train.parquet", "rb")。 它们完全不知道、也不需要知道底层是什么磁盘。Block 设备必须先格式化才能有文件系统，而在容器里做 mkfs 需要特权模式，并且每次 Pod 重启都需要重新挂载，这不是一个合理的生产方案。

正确的解决方向

去问 ACS 提供本地盘 PVC 的团队，本地盘有两种暴露方式：

模式volumeMode容器得到什么适用场景
Block 模式 local-passthrough Block /dev/xvdb 裸块设备 数据库 direct I/O、Ceph
Filesystem 模式 常规本地盘 Filesystem /mnt/cache 目录 AI 训练、JuiceFS 缓存、所有常规应用 ✅
💡 向 ACS 团队要求： 提供一个类似 alicloud-local-disk / alicloud-local-ssd 这类 Filesystem 模式的 StorageClass，或者直接提供 volumeMode: Filesystem 的 PVC。

备选方案：直接用 HostPath

如果 ACS 在宿主机上也将本地盘挂载到了某个路径（如 /mnt/nvme0）， 可以完全绕开 PVC 问题，直接用 type: HostPath，这也是最简单稳定的方案： 

cacheDirs:
  - type: HostPath
    path: /mnt/nvme0   # 宿主机上已格式化挂载好的本地盘路径
# 确认宿主机上的挂载点
df -h | grep nvme
lsblk -o NAME,MOUNTPOINT,SIZE | grep -v loop

用 HostPath 时不需要 enableScheduling，Operator 默认行为就是一个 Node 一个 Worker，配置最简洁。

4Filesystem 模式 PVC + 完整 YAML

假设 ACS 提供了 Filesystem 模式的本地盘 StorageClass（如 alicloud-local-ssd）， 此时 PVC 可以正常被 Operator 的 cacheDirs 使用。

⚠️ 仍然需要 enableScheduling: true： 即使是 Filesystem 模式，Local PV 上依然有 nodeAffinity（只能被该 Node 的 Pod 使用）。 不开 enableScheduling 时 Operator 自己的调度逻辑不感知这个约束，可能导致调度失败。 开启后需配合反亲和性，保证每台 Node 最多一个 Worker。

完整 YAML 示例

## 1. JuiceFS 认证 Secret
apiVersion: v1
kind: Secret
metadata:
  name: juicefs-secret
  namespace: juicefs-cache-group
type: Opaque
stringData:
  name: your-volume-name
  token: your-token
  access-key: your-ak
  secret-key: your-sk
  # 私有化部署加这行，公有云可删
  # envs: '{"BASE_URL": "http://<console-ip>:8080/static"}'

---
## 2. CacheGroup 主体
apiVersion: juicefs.io/v1
kind: CacheGroup
metadata:
  name: ppu-cache-group
  namespace: juicefs-cache-group
spec:
  secretRef:
    name: juicefs-secret

  # 自定义缓存组名，不填默认是 ${namespace}-${name}
  cacheGroup: ppu-cache-group

  # 开启 K8s 标准调度器 —— Local PVC 必须开启（让调度器感知 PV 的 nodeAffinity）
  enableScheduling: true

  # 节点缩容时先迁移缓存再删节点
  waitingDeletedMaxDuration: 1h

  # 删除节点时清理本地缓存
  cleanCache: true

  worker:
    template:
      # 只在打了这个标签的节点上部署 Worker
      nodeSelector:
        juicefs.io/cg-worker: "true"

      image: juicedata/mount:ee-5.1.1-1faf43b   # 换成实际版本

      # 反亲和性：保证每台 Node 最多一个 Worker Pod
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: juicefs.io/cache-group
                    operator: In
                    values:
                      - ppu-cache-group   # 与 spec.cacheGroup 保持一致
              topologyKey: "kubernetes.io/hostname"

      opts:
        - cache-size=204800         # 200GB，按实际盘大小调整
        - free-space-ratio=0.05
        - group-weight=100
        - buffer-size=4096

      # 使用 Filesystem 模式的本地盘 PVC
      cacheDirs:
        - type: PVC
          name: local-ssd-pvc-node-ppu-01   # 该节点对应的 PVC 名

      resources:
        requests:
          cpu: 500m
          memory: 512Mi
        limits:
          cpu: "4"
          memory: 4Gi

    # 不同节点可能 PVC 名字不同，用 overwrite 为每个节点单独指定
    overwrite:
      - nodes:
          - node-ppu-01
        cacheDirs:
          - type: PVC
            name: local-ssd-pvc-node-ppu-01
      - nodes:
          - node-ppu-02
        cacheDirs:
          - type: PVC
            name: local-ssd-pvc-node-ppu-02
      - nodes:
          - node-ppu-03
        cacheDirs:
          - type: PVC
            name: local-ssd-pvc-node-ppu-03

---
## 3. 应用侧 StorageClass —— 挂载时加入缓存组
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: juicefs-sc
mountOptions:
  - cache-group=ppu-cache-group
  - no-sharing   # 业务客户端加入缓存组但不参与构建，避免频繁创销影响命中率

部署命令

# 1. 给 PPU 节点打标签
kubectl label node node-ppu-01 node-ppu-02 node-ppu-03 juicefs.io/cg-worker=true

# 2. 创建 namespace
kubectl create namespace juicefs-cache-group

# 3. 应用配置
kubectl apply -f juicefs-cache-group.yaml

# 4. 查看状态（等待 PHASE=Ready）
kubectl get cachegroups -n juicefs-cache-group -w

# 5. 确认每台 Node 都有一个 Worker Pod
kubectl get pods -n juicefs-cache-group -o wide

# 确认 PVC 名字（通过 nodeAffinity 反查）
kubectl get pv -o json | python3 -c "
import json, sys
pvs = json.load(sys.stdin)['items']
for pv in pvs:
    na = pv.get('spec', {}).get('nodeAffinity', {})
    terms = na.get('required', {}).get('nodeSelectorTerms', [{}])
    node = ''
    for t in terms:
        for e in t.get('matchExpressions', []):
            if e['key'] == 'kubernetes.io/hostname':
                node = e['values'][0]
    claim = pv.get('spec', {}).get('claimRef', {}).get('name', '')
    if claim:
        print(f'Node: {node:30s}  PVC: {claim}')
"
✅ 如果嫌麻烦，用 HostPath 最简单： 如果 ACS 宿主机上本地盘已有挂载点， 完全不需要 enableScheduling，也不需要反亲和性，直接 cacheDirs: [{type: HostPath, path: /mnt/nvme0}]，Operator 默认就是一 Node 一 Worker。
