JuiceFS O_DIRECT 读优化实践

通过 O_DIRECT + preadv 将 JuiceFS 顺序读吞吐从 1.3 GB/s 提升至 8.1 GB/s

2026-06-06 · POC 性能调优

1. I/O 模式分析

测试脚本 test.py 模拟量化交易场景中的随机批量读取，核心 I/O 模式如下：

数据布局

  • X 特征矩阵：每个样本 15926 × 1800 × 4 = ~109 MB（float32）
  • Y 标签向量：每个样本 15926 × 4 = ~62 KB（float32）
  • 按日期分目录，每个日期对应一个连续二进制文件

读取模式

  • 每次迭代随机选日期 + 随机时间偏移，pread 定位到文件中特定 offset
  • batch_size=8，iters=50，共 400 次读取
  • 单次读 109 MB，总数据量 ~42.8 GB
  • 典型的大块随机读（large random read）模式

性能瓶颈分析

关键发现：Python 标准 open("rb") 使用 buffered I/O，即使 JuiceFS 挂载设置了 --max-write=1M， 内核仍会将读请求拆分为 128KB 的 FUSE 请求。每次 109MB 的读取被拆分成 ~870 个 FUSE 请求， 带来大量的内核态/用户态上下文切换开销。

# 标准 buffered I/O：每次 read() 被内核拆成 128KB FUSE 请求
for d in dates:
    fds[(d, "x")] = open(xfile, "rb")
    fds[(d, "y")] = open(yfile, "rb")

# 读取：seek + read，每次分配新的 bytes 对象
fds[(d, "x")].seek(ti * x_bytes)

3. 调优内容

核心改动两点：

a) JuiceFS：启用 O_DIRECT 模式

使用 os.O_DIRECT 标志打开文件，绕过内核 page cache。这样：

  • FUSE 请求大小不再受 128KB 限制，--max-write=1M 参数生效
  • 单次 109MB 读取只需 ~109 个 FUSE 请求（vs 之前的 ~870 个）
  • 大幅减少内核态/用户态上下文切换
  • 配合预分配 buffer（bytearray）+ preadv，避免每次读取分配 109MB 内存

b) 本地 NVMe：自动回退到 Buffered I/O

本地块设备的 O_DIRECT 要求内存地址按扇区大小对齐（通常 512B 或 4KB）， Python 的 bytearray 不保证对齐，直接调用会报 EINVAL。 脚本通过 probe 机制自动检测：

  • 先尝试 O_DIRECT 读一次
  • 如果失败（OSError），自动回退到普通 buffered I/O
  • NVMe 仍使用原有方式，不影响基线对比

关键代码变更

# 1. 用 O_DIRECT 打开文件（绕过内核 page cache）
fds[(d, "x")] = os.open(xfile, os.O_RDONLY | os.O_DIRECT)

# 2. 预分配 buffer，避免每次读分配 109MB
x_buf = bytearray(x_bytes)  # ~109MB，分配一次
y_buf = bytearray(y_bytes)  # ~62KB

# 3. 用 preadv 读入已有 buffer（零分配）
def do_read(key, buf, offset):
    return os.preadv(fds[key], [buf], offset)

# 4. Probe：O_DIRECT 失败则回退到 buffered I/O
try:
    do_read((dates[0], "x"), x_buf, 0)
except OSError:
    logging.info("O_DIRECT not supported, falling back")
    for key in fds:
        os.close(fds[key])
        fds[key] = os.open(paths[key], os.O_RDONLY)
    def do_read(key, buf, offset):
        return len(os.pread(fds[key], len(buf), offset))

要点总结

  • JuiceFS + O_DIRECT 读吞吐提升 6.3 倍（1,279 → 8,083 MiB/s），达到缓存集群带宽上限
  • 优化后 JuiceFS 吞吐是本地 NVMe 单盘的 4.3 倍，充分体现分布式缓存聚合带宽优势，当然NVMe也可以用DIO模式不过需要内存对齐，这里就不做对应优化。
  • 代码改动极小（~15 行），对应用逻辑零侵入
  • 自动检测 O_DIRECT 支持，本地盘自动回退，一套代码兼容所有存储
多节点横向扩展：JuiceFS 缓存集群的带宽可随节点数线性增长。当前测试为 6 节点缓存集群（每节点 2×3.5T NVMe）， 单客户端已达到 ~8 GB/s 读带宽。多客户端并发读取时，聚合带宽可进一步提升。
